    Naming Conventions__=================================================================
    Prioritize internal consistency
    Names that are visible to the user as public parts of the API should follow conventions that reflect usage rather than implementation.

    _single_leading_underscore: weak â€œinternal useâ€ indicator. E.g. from M import * does not import objects whose names start with an underscore.
    single_trailing_underscore_: used by convention to avoid conflicts with Python keyword.
    __double_leading_underscore: when naming a class attribute, invokes name mangling (inside class FooBar, __boo becomes _FooBar__boo; see below).
    __double_leading_and_trailing_underscore__: â€œmagicâ€ objects or attributes that live in user-controlled namespaces. E.g. __init__, __import__ or __file__. Never invent such names; only use them as documented.

    Avoid these names: 'I', 'O', 'l' and use 'L' instead if necessary
